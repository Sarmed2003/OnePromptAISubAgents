# Swarm lane 10: DynamoDB Table Provisioning

## Overview

Lane 10 is the DevOps agent responsible for provisioning the DynamoDB table that serves as the persistent backend for fossil specimen and bone measurement data in the DinoLab system. This lane executes infrastructure-as-code through a SAM (Serverless Application Model) template to create a production-ready DynamoDB table with appropriate schema, indexing, and access patterns.

## Responsibilities

The DevOps agent in lane 10 performs the following core functions:

- **Infrastructure Provisioning**: Deploys the DynamoDB table using the SAM template located at `dinolab/infra/template.yaml`
- **Schema Configuration**: Establishes the table schema with `specimen_id` as the partition key and `bone_id` as the sort key, enabling efficient queries on fossil specimens and their associated bone measurements
- **Global Secondary Index (GSI) Setup**: Configures at least one GSI to support alternative query patterns beyond the primary key structure
- **Backend Integration**: Ensures compatibility with the DynamoDB backend implementation in `oneprompt/db.py`

## Input Contract

Lane 10 receives a `validation_report` from upstream lanes containing validation results and configuration parameters necessary to provision the table with appropriate settings and capacity.

## Output Contract

Lane 10 outputs a `table_ready` signal that confirms successful DynamoDB table provisioning and readiness for data operations. This signal includes:

- Confirmation that the table exists and is in ACTIVE status
- Verification that the partition key (specimen_id) and sort key (bone_id) are properly configured
- Confirmation that all GSIs are deployed and queryable
- Endpoint information for downstream consumers

## Downstream Consumers

- **Lane 11 (Security)**: Consumes the `table_ready` signal to apply security policies, encryption settings, and access controls to the provisioned table
- **Lane 31 (Auditor)**: Consumes the `table_ready` signal to establish audit logging and compliance monitoring on the DynamoDB table

## Technical Details

The table provisioning follows infrastructure-as-code best practices:

- All configuration is defined declaratively in the SAM template
- The template is version-controlled and reproducible across environments
- Capacity settings and throughput are optimized for the expected query patterns
- The table integrates seamlessly with the Python DynamoDB client in `oneprompt/db.py`
